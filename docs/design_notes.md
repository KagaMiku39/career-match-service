# Design Notes

## 为什么做这个项目

目标岗位是大模型应用后端开发实习生，JD 里重点提到：

- Prompt 工程
- 大模型 API 集成
- 后端 API
- 数据库对接
- Python/Java
- 稳定性和可维护性

所以这个项目选择做“简历与岗位 JD 匹配分析”这个场景。它足够小，适合快速完成；同时又能覆盖 API、数据库、Prompt、LLM 调用和错误处理这些后端能力。

## 为什么不是一开始全靠大模型

如果项目完全依赖大模型 API，会有几个问题：

- 没有 API Key 时无法演示。
- 网络、额度、模型返回格式都可能导致失败。
- 面试时很难解释模型内部到底怎么分析。

所以第一版先做本地规则分析：

```text
JD 提到某关键词，简历也提到 -> matched
JD 提到某关键词，简历没提到 -> missing
```

规则模式提供一个可解释 baseline。配置模型 API 后，再切换到 LLM 模式。

## 为什么要保存历史记录

只返回即时分析结果像一个函数 demo。

保存历史记录后，服务变成一个更完整的后端应用：

```text
接收请求
处理业务
持久化结果
查询历史
```

这也是数据库接入的最小闭环。

## 为什么用 SQLite

SQLite 不需要单独启动数据库服务，适合本地 demo 和学习。

这个项目的表结构和 SQL 操作比较简单，后续迁移到 MySQL 时主要需要替换 `storage.py`，接口层和业务层可以保持相对稳定。

## 后续计划

- 增加 MySQL 版本的 storage adapter。
- 增加 Redis 缓存，缓存重复简历/JD 的分析结果。
- 增加 PDF 简历解析。
- 增加 RAG：根据简历、项目说明和岗位 JD 检索相关片段后再调用模型。
- 增加 Dockerfile 和 docker-compose，降低部署成本。
